Overview

CyberClaw holds a local model catalog tracking pricing values per million tokens (input/output), context windows, capabilities (reasoning, vision, tool use), status (available/preview/deprecated), and auto-replacements.

Common Catalog Code Operations

Query or find appropriate models in Python using these simple operations:

from cyberclaw.core.model_catalog import get_catalog

# Initialize central registry
catalog = get_catalog()

# Estimate token usage cost in USD
cost = catalog.estimate_cost("openai", "gpt-4o-mini", input_tokens=1500, output_tokens=800)
print(f"Call Cost: ${cost:.6f} USD")

# Query the cheapest model that supports vision
cheapest_vision_model = catalog.find_cheapest(needs_vision=True)
print(f"Selected Model: {cheapest_vision_model.ref}")

Smart Gateway Routing & Rate-Limit Failover

CyberClaw features an intelligent, multi-provider model routing layer. If a high-priority provider encounters rate limits (429) or server failures, CyberClaw transparently routes the query to backup/secondary providers with no downtime.

🛡️ SMART ROUTING KEY CAPABILITIES:

  • Circuit-Breaker Cooling-Off: Automatically applies cooling-off periods (120s for 429 rate-limit errors, 60s for standard server errors) to avoid hammering failing endpoints.
  • Equal-Priority Load Balancing: Traffic is automatically shuffled and balanced across multiple configured providers sharing the same priority level.
  • Zero-Downtime Failover: Transparently falls back to backup providers in order of priority level (sorted lower to higher priority number) if primary models fail.

Configure multiple LLM provider entries in your config.user.yaml to enable automatic failovers:

llm:
  enable_failover: true
  providers:
    - id: primary-a
      provider: openai
      model: gpt-4o
      api_key: "sk-..."
      priority: 1
      enabled: true
    - id: primary-b
      provider: groq
      model: llama-3.3-70b-versatile
      api_key: "gsk-..."
      priority: 1
      enabled: true
    - id: backup-1
      provider: gemini
      model: gemini-2.5-pro
      api_key: "AIzaSy..."
      priority: 2

Task System & Database Schema

Master background task workers and explore the ACID-protected SQLite schema holding tasks and conversation histories.

Database Locations

CyberClaw holds two dedicated database directories under your workspace folder:

  • History Store: workspace/history.db
  • Task Registry: workspace/.tasks/tasks.db

The SQLite Task Schema

Below is the exact database schema configuration defining background tasks:

CREATE TABLE tasks (
    id TEXT PRIMARY KEY,
    name TEXT NOT NULL,
    agent_id TEXT NOT NULL,
    instruction TEXT NOT NULL,
    status TEXT NOT NULL DEFAULT 'pending',
    priority TEXT NOT NULL DEFAULT 'normal',
    result TEXT DEFAULT '',
    error TEXT DEFAULT '',
    created_at REAL NOT NULL,
    started_at REAL DEFAULT 0,
    completed_at REAL DEFAULT 0,
    timeout_seconds INTEGER DEFAULT 300,
    retry_count INTEGER DEFAULT 0,
    max_retries INTEGER DEFAULT 1,
    metadata TEXT DEFAULT '{}'
);

Running a Task Programmatically

from cyberclaw.core.tasks import TaskStore, TaskExecutor
from pathlib import Path

# Connect to database and submit tasks
store = TaskStore(Path("./workspace/.tasks/tasks.db"))
executor = TaskExecutor(store, max_concurrent=3)

task = await executor.submit(
    name="Log analyzer",
    agent_id="cyberclaw",
    instruction="Analyze production error records from database"
)
print(f"Task submitted: {task.id}")

Context Engine

Discover how CyberClaw dynamically builds token-budgeted prompt payloads using plugin-contributed data feeds.

Dynamic Context Injection

The ContextEngine automatically collects context updates from multiple sources, sorts them by priority, and ensures the payload fits perfectly inside the context window constraint before querying the LLM provider.

Registering a Custom Context Feed

Add a new system context provider in your backend plugins:

from cyberclaw.core.context_engine import ContextEngine, ContextRequest, ContextContribution

engine = ContextEngine()

# Register a prioritized feed that injects the current timestamp
def system_time_provider(request: ContextRequest):
    import time
    return ContextContribution(
        source="time_agent",
        priority=100, # High priority, injected first
        content=f"Current System Time: {time.strftime('%Y-%m-%d %H:%M:%S')}"
    )

engine.register_provider("time", system_time_provider)

Holographic Memory VSA

Dive deep into CyberClaw's two-phase long-term memory engine utilizing circular Vector Symbolic Architecture (VSA / HRR) and SQLite FTS5.

The Challenge of AI Memory

Traditional AI agents rely on simple raw chat history window limits or flat vector embeddings stored in complex external vector databases. This leads to either catastrophic forgetting (when history gets trimmed) or slow, expensive indexing processes.

CyberClaw solves this with a **Two-Phase Holographic Memory Consolidation Engine**:

  1. Phase 1: Real-time Rollout & Vector Encoding — Conversational facts are extracted at runtime, projected into a circular Vector Symbolic Architecture, and stored in a local SQLite FTS5 database for instant, sub-millisecond retrieval.
  2. Phase 2: Local Git Consolidation — Periodically, stale or duplicate facts are consolidated, written into structured MEMORY.md directories, and compiled into active cognitive skill templates within a local Git repository.

Circular Vector Symbolic Architecture (VSA)

Holographic Reduced Representations (HRR) utilize circular phase vectors to perform algebraic operations directly on data vectors. Instead of sending raw text embeddings to an external database, CyberClaw binds semantic relations directly:

Vector Binding:   C = A ⊛ B  (Circular Convolution)
Vector Bundling:  S = X + Y  (Vector Addition)
Vector Unbinding: B' = C ⊛ A⁻¹ (Approximate Inverse Retrieval)

This mathematical structure allows CyberClaw to "bundle" hundreds of facts into a single dense circular vector representation, executing blazing-fast associative memory scans locally on your CPU with zero external API calls.

SQLite Memory Schema

Consolidated facts are persisted locally in workspace/history.db. Below is the active schema representing the facts store:

CREATE TABLE memory_facts (
    id TEXT PRIMARY KEY,
    session_id TEXT NOT NULL,
    fact_category TEXT NOT NULL,
    fact_content TEXT NOT NULL,
    vsa_vector BLOB NOT NULL, -- Circular phase float vector array
    occurrence_count INTEGER DEFAULT 1,
    last_updated REAL NOT NULL
);

-- Virtual table for high-speed full-text search (FTS5)
CREATE VIRTUAL TABLE memory_facts_fts USING fts5(
    fact_category,
    fact_content,
    content='memory_facts',
    content_rowid='rowid'
);

Writing a Memory Adapter

POSIX Windows Shell Bridge

Learn how CyberClaw translates Unix shell syntax, redirection targets, and slash paths to Windows natively and securely.

Why Cross-Platform Scripting Fails

AI agents frequently write scripts or run bash commands (e.g. rm -rf tmp/ && mkdir tmp or cat /etc/hosts). When executed on a Windows host machine, these commands instantly crash because Windows command processors (PowerShell / cmd.exe) do not recognize Unix syntax, forward slashes, or system files like /dev/null.

CyberClaw solves this with a built-in POSIX Windows Shell Bridge that intercepts, parses, and translates Unix parameters before they reach the execution runner.

Syntax Translation & Command Mappings

The bridge translates Unix commands programmatically according to this rigorous spec:

Unix Command / Syntax Windows Translation Description
/dev/null nul Translates standard stream redirection targets
/c/Users/Sourov/project C:\Users\Sourov\project Maps absolute drive letters and forward slashes
open file.txt / xdg-open start "" file.txt Translates system file opener calls
ls -la dir Maps directory listings programmatically
cat file.log type file.log Maps stream prints for file inspection
touch file.txt type nul > file.txt Creates empty files cleanly on Windows cmd/powershell
rm -rf folder rmdir /s /q folder Translates directory cleanup chains recursively

How It Works under the Hood

When the AI triggers a bash or powershell tool, the command is processed by CyberClaw's translation pipeline:

Step 1: Detection

CyberClaw checks the host operating system. If the host is Windows, the translation bridge is engaged.

Step 2: String Parsing

Regex tokenizers parse variables, paths, and pipe chain segments. It matches patterns like /([a-zA-Z])/(.*) to resolve drive letters (e.g. /d/project ➔ D:\project).

Step 3: Safe Subprocess Execution

The translated command is wrapped inside a secure Win32 Job Object sandbox (restricting memory to 50MB) and executed safely.

Declaring Robust Cross-Platform Tools

When developing custom plugins for CyberClaw, write tool declarations using the built-in translation APIs to ensure your plugin is immediately cross-platform:

from cyberclaw.security.sandbox import execute_command_in_sandbox
from cyberclaw.utils.posix_bridge import translate_command

# Command input written in POSIX style
posix_cmd = "cat /c/Users/Sourov/Desktop/logs/error.log > /dev/null"

# Translate command automatically based on OS
translated_cmd = translate_command(posix_cmd)
print("Running command:", translated_cmd)
# Output on Windows: "type C:\\Users\\Sourov\\Desktop\\logs\\error.log > nul"

# Execute safely inside sandbox
result = await execute_command_in_sandbox(translated_cmd, timeout_seconds=30)
if result.exit_code == 0:
    print("Execution Success:", result.stdout)
else:
    print("Execution Failed:", result.error)
